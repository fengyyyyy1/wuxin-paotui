# Codex 协作规范（CODEX_RULE）

本规范用于约束 Codex 在本项目中的协作方式。

## 一、回复规范

1. 每次回复用户时，第一句话必须固定为：

你好悠悠球。

2. 回复必须简洁、准确，不重复无意义内容。

3. 每次开发完成后，必须按照固定格式汇报。

4. 不得伪造接口、构建、测试、数据库或人工验收结果；未通过项必须明确记录。

----------------------------------------

## 二、开发前要求

每次开始开发之前，必须先阅读：

docs/PROJECT_STATUS.md

确认：

- 当前项目版本
- 当前开发断点
- 已完成模块
- 下一步开发计划

不得重复开发已经完成的功能。

不得跳过当前开发计划。

----------------------------------------

## 三、开发规范

开发过程中必须遵守：

docs/DEVELOPMENT_RULE.md

包括但不限于：

- 分层规范
- 命名规范
- API规范
- 数据库规范
- 文档规范
- Git规范

----------------------------------------

## 四、开发完成后必须说明

必须明确回复：

1. 新增了哪些文件

2. 修改了哪些文件

3. 是否修改数据库

4. 是否新增 SQL

5. 是否修改 Maven

6. 是否启动项目

7. 是否调用端口

8. 是否执行 Compile

9. Compile 是否成功

10. Postman 如何测试

11. Navicat 如何验证

----------------------------------------

## 五、文档同步

如果开发涉及：

- 新接口
- 数据库
- 开发流程
- 测试流程
- 当前开发断点

必须同步更新：

docs/API.md

docs/DATABASE.md

docs/CHANGELOG.md

docs/PROJECT_STATUS.md

docs/POSTMAN.md

必要时更新：

docs/README.md

docs/TEST_ACCOUNT.md

docs/TEST_ENVIRONMENT.md

测试账号、测试数据 ID、Token 获取方式或完整测试时间发生变化时，必须更新 `TEST_ENVIRONMENT.md`。

----------------------------------------

## 六、数据库规范

任何数据库修改必须明确说明：

- 修改了哪些字段
- 修改了哪些索引
- 是否新增 SQL
- SQL 文件路径

不得隐瞒数据库修改。

数据库DDL只能由用户或维护人员按增量SQL人工执行，Codex不得自动执行。

----------------------------------------

## 七、Git规范

1. Git提交必须使用中文标签和与实际修改一致的说明。
2. 提交前必须检查Build、Lint、TypeScript、Compile、自动化测试和工作区范围。
3. 不得把无关工作区修改混入提交，不得使用强制推送。
4. 只有数据库、接口、构建、文档、推送和干净工作区条件全部满足时才能创建版本标签。

----------------------------------------

## 八、禁止事项

禁止：

- 未说明就启动项目
- 未说明就调用端口
- 未说明就修改数据库
- 删除已有业务功能
- 修改项目规范文档而不说明

----------------------------------------
