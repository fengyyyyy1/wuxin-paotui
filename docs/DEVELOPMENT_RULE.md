# 开发规范

> 项目：五鑫跑腿（Wuxin Paotui）  
> 当前版本：V1.9 总控后台第一阶段

## 一、分层规范

1. Controller 不写业务。
2. Service 写业务。
3. Mapper 只负责数据库。
4. Entity 不直接返回前端。
5. 统一使用 DTO 接收请求参数。
6. 统一使用 VO 返回前端数据。
7. 统一使用 Result 作为接口返回结构。
8. 统一使用 BusinessException 表示业务异常。

## 二、接口开发规范

新增接口必须更新：

- `API.md`
- `CHANGELOG.md`
- `PROJECT_STATUS.md`
- `POSTMAN.md`

接口开发要求：

- Controller 只接收参数、调用 Service、返回 Result。
- Service 负责业务校验、权限校验、事务、状态流转。
- Mapper 不写业务逻辑。
- 不直接返回 Entity。
- 不返回敏感字段。
- API 文档中的请求方式和路径必须以 Controller 实际映射为准。
- 每次版本收尾必须逐个检查所有 Controller 与 `API.md`。
- 文档不得记录当前 Controller 不存在的接口。
- 四端闭环联调中发现的断点只允许接入或修复已有业务链路，不借机开发无关新功能。

## 三、数据库变更规范

新增数据库字段必须：

- 更新 `DATABASE.md`
- 新增 SQL 升级脚本

禁止：

- 修改历史 SQL
- 启动程序时自动修改数据库
- 删除已有业务表
- 重建已有业务表
- 未确认字段含义时随意映射 Entity

数据库字段要求：

- Entity 和 Mapper 映射必须以真实数据库结构为准。
- 历史聊天记录、旧文档和字段命名推测不能替代数据库核对。
- 发现代码与数据库不一致时，应先记录风险并设计增量迁移。
- 本地真实HTTP或数据库联调必须注入`DB_PASSWORD`；缺少凭证时不得宣称数据库回归通过。

## 四、异常规范

所有异常统一通过：

- `BusinessException`
- `GlobalExceptionHandler`

要求：

- 可预期业务异常不能返回 500。
- 未知系统异常统一返回 500。
- 不向前端暴露堆栈信息。

## 五、验收规范

开发完成必须执行：

1. Compile
2. Postman
3. Navicat
4. Docs

测试要求：

- 必须测试正常流程。
- 必须测试异常流程。
- 涉及用户数据时必须测试越权访问。
- 不得把未实际执行或未通过的测试写成通过。
- 测试账号统一维护在 `TEST_ACCOUNT.md`。
- 测试环境和业务数据 ID 统一维护在 `TEST_ENVIRONMENT.md`。
- 测试账号密码或测试数据变化后必须立即同步文档。

## 六、版本收尾规范

每完成一个版本必须更新：

1. `README.md`
2. `API.md`
3. `DATABASE.md`
4. `PROJECT_STATUS.md`
5. `CHANGELOG.md`
6. `TEST_ACCOUNT.md`
7. `TEST_ENVIRONMENT.md`

版本完成后：

1. 确认 Compile、Postman、Navicat 和权限测试结果。
2. 更新项目状态和当前开发断点。
3. 使用中文标签提交Git。
4. 仅为已经验收通过的稳定版本创建 Git Tag。

## 七、依据优先级

项目事实依据按以下顺序确认：

```text
真实数据库
↓
当前代码
↓
当前项目文档
↓
历史记录
```

接口路径必须以 Controller 为准，数据库字段必须以真实数据库为准。

## 八、固定要求

以后所有开发必须遵守本规范。

### 总体路线与版本边界

1. `ROADMAP.md`是项目唯一总体路线，其他文档不得维护冲突的完整路线。
2. 新功能开发前必须先确认所属目标版本。
3. 不允许因临时讨论扩大当前版本范围。
4. 小程序以核心业务闭环为完成标准，不以页面数量为目标。
5. 非必要功能必须进入后续候选清单，不得插入当前版本。
6. 所有数据库变更必须通过增量脚本管理。
7. Codex不得自动执行用户本地数据库或生产数据库升级脚本。
8. 未经用户明确要求，Codex不得自动执行Git Commit或Git Push；用户明确要求时仍禁止强制推送。
9. 数据库密码、密钥等敏感配置必须使用环境变量。
10. 每个版本必须完成编译、自动化测试、Postman、Navicat和文档验收。
11. Git提交前必须检查`git status`、`git diff`、`git diff --cached`、敏感信息和构建产物。
12. 测试账号、测试数据ID和状态变化必须同步到测试文档，不记录长期Token、AppSecret、私钥或证书。

## 九、支付安全规范

1. 支付金额只能读取数据库并由服务端计算，前端不得提交最终支付金额。
2. 订单履约状态`status`与支付状态`pay_status`必须保持独立。
3. 创建预支付单不得直接把订单标记为已支付。
4. 只有可信支付回调或主动查单结果可以调用统一支付确认事务。
5. 小程序`wx.requestPayment`成功回调不能作为服务端支付成功依据。
6. 微信回调必须使用原始请求体，通过官方SDK验签、解密并校验商户、订单和金额。
7. 重复支付通知不得重复更新订单、重复写订单日志或重复记账。
8. 私钥、APIv3密钥、证书和本地真实配置禁止提交Git。
9. Mock支付和旧模拟支付接口必须由环境配置控制，生产环境默认关闭。
10. 日志不得输出私钥、APIv3密钥、完整回调敏感数据或用户身份敏感信息。

## 十、微信小程序登录安全规范

1. 小程序登录接口只接收`wx.login`临时code，不接收openid、unionid、session_key或userId。
2. AppSecret只能通过环境变量或未跟踪配置注入。
3. code不得缓存或持久化，session_key不得持久化、返回前端或写入日志。
4. 日志不得输出完整AppSecret、code、openid、unionid或session_key。
5. 真实code2session调用必须设置连接和读取超时。
6. 微信网络调用必须位于数据库事务之外，本地用户查询和创建使用短事务。
7. openid查询必须过滤逻辑删除用户，禁用用户不得登录。
8. 首次自动注册必须依靠`uk_openid`处理并发，不得仅依赖先查后插。
9. 不得根据unionid、手机号或昵称自动合并账号。
10. Mock微信登录默认关闭，`prod` Profile禁止启用。

## 十一、Git提交规范

1. Git Commit Message统一使用中文。
2. 禁止使用`feat:`、`fix:`、`docs:`等英文Commit前缀。
3. 提交信息必须能够直接反映本次开发内容。
4. 一个Commit应对应一个完整功能。
5. 功能完成人工验收并完成文档审计后才能提交。

统一格式：

| 类型 | 格式 | 示例 |
| --- | --- | --- |
| 新增功能 | `完成：模块-功能` | `完成：用户模块-微信登录` |
| 修复问题 | `修复：问题说明` | `修复：BCrypt随机密码超过72字节` |
| 文档更新 | `文档：更新内容` | `文档：更新PROJECT_STATUS` |
| 数据库 | `数据库：变更内容` | `数据库：新增订单评价表` |
| 重构 | `重构：重构内容` | `重构：支付网关抽象` |

提交前必须确认以下文档与实际开发一致：

1. `README.md`
2. `PROJECT_STATUS.md`
3. `CHANGELOG.md`
4. `API.md`
5. `POSTMAN.md`
6. `DATABASE.md`
7. `TEST_ACCOUNT.md`
8. `TEST_ENVIRONMENT.md`

## V1.9 总控后台规则

1. 所有 Admin Controller 使用 `@AdminPermission` 权限点，不再只判断单一 ADMIN 角色。
2. 运营金额、比例、距离、超时、首页内容必须落库，禁止新增前端硬编码业务配置。
3. 敏感配置不得通过公共接口下发，后台查询必须掩码，日志不得记录明文密钥。
4. 管理员写操作必须写入 `admin_operation_log`；订单状态变化同时写入既有 `order_log`。
5. 数据库只新增编号递增的增量SQL，禁止编辑历史脚本，禁止应用自动执行生产迁移。
6. Admin 页面必须同时校验路由权限和操作按钮权限，后端权限校验始终是最终边界。
7. 新接口至少通过 Java 21 Compile、Mapper 上下文解析和真实数据库 Postman 回归后才能标记验收通过。
